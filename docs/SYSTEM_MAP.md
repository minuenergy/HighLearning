# SocraTeach System Map

Last updated: 2026-04-12  
Status: Product and architecture source of truth

이 문서는 SocraTeach의 목표, 사용자 경험, 핵심 기능, 권한 규칙, 데이터 원칙, 시스템 구조를 한 번에 정의하는 기준 문서입니다.

문서 운영 규칙:
- 제품 목표나 주요 화면 구조가 바뀌면 이 문서를 먼저 갱신합니다.
- 현재 구현 범위와 우선순위는 `docs/IMPLEMENTATION_STATUS.md`에서 함께 관리합니다.
- DB 구조 변경과 적용 순서는 `docs/DB_MIGRATIONS.md`에서 함께 관리합니다.
- 새로운 페이지, API, 마이그레이션, AI 파이프라인이 생기면 두 문서를 함께 갱신합니다.

## 0. DB 마이그레이션 운영 원칙

- 새로운 기능이 DB 테이블, 컬럼, 인덱스, 정책을 요구하면 반드시 마이그레이션 파일로 기록합니다.
- Fresh setup은 `backend/supabase/migrations/001`부터 최신 파일까지 순서대로 적용합니다.
- 기존 프로젝트는 아직 적용하지 않은 뒤 번호 마이그레이션만 추가 적용합니다.
- 현재 워크스페이스 개편 기능은 `006_workspace_domain_and_settings.sql`, `007_verification_and_material_jobs.sql`, `008_material_pages_and_generation.sql` 적용을 전제로 합니다.
- 기본 개발 기준 DB/Auth 런타임은 `외부 Supabase`가 아니라 `backend/supabase/config.toml` 기반의 `local-only Supabase` 입니다.
- 구체적인 적용 절차와 확인 방법은 `docs/DB_MIGRATIONS.md`를 기준으로 합니다.

## 1. 제품 목적

SocraTeach의 목적은 "AI가 학생을 대신 가르치는 서비스"가 아니라, 교사가 실제 수업을 더 잘 운영하도록 돕는 "교사 보조형 학습 운영 시스템"을 만드는 것입니다.

핵심 목표:
1. 교사가 실제로 쓰는 교재 PDF를 업로드하면 과목별, 교재별, 목차별로 구조화한다.
2. 구조화된 교재 내용을 근거로 시험지와 숙제를 자동 생성한다.
3. 교사는 자동 생성된 시험지를 직접 검토하고 수정, 삭제, 추가, 저장, 예약 배포할 수 있다.
4. 학생은 온라인으로 시험을 보고 자동 채점 결과를 받는다.
5. 학생은 오답에 대해 소크라테스식 AI 튜터와 대화하면서 정답을 스스로 도출한다.
6. 학생이 처음부터 맞혔거나, 복기 후 최종 정답에 도달했을 때만 해설과 교재 근거를 본다.
7. 교사는 반별, 과목별, 목차별, 문항별 오답률과 튜터 대화 데이터를 함께 분석한다.
8. 시스템은 교사가 보강수업에서 실제로 말할 내용, 다시 보여줄 교재 범위, 추가 문제 방향까지 제안한다.

## 2. 제품 원칙

### 2-1. 교재 근거 우선

- 모든 문제는 반드시 특정 교재, 특정 목차, 특정 페이지 범위와 연결되어야 합니다.
- 문제, 정답, 해설, 교육 목적은 모두 시험 DB에 메타데이터로 저장되어야 합니다.
- 문제를 수정하더라도 "어느 교재 내용을 근거로 했는지" 연결은 유지되어야 합니다.

### 2-2. AI는 힌트 역할

- AI 튜터는 학생에게 정답을 바로 주지 않습니다.
- AI는 학생이 틀린 문제를 이해하도록 질문, 힌트, 사고 유도만 제공합니다.
- 학생이 최종 정답을 맞히기 전까지 정답/해설/근거 원문은 잠금 상태를 유지합니다.

### 2-3. 교사 중심 UX

- 모든 핵심 화면은 "교사가 무엇을 하고 싶은가"를 기준으로 설계합니다.
- 기능을 한 페이지에 다 때려 넣지 않고, 선택 후 팝업 또는 전용 작업 공간에서 집중해서 처리하게 합니다.
- 그래프와 캘린더는 설명용이 아니라 실제 수업 의사결정 도구여야 합니다.

### 2-4. 학생 경험은 단순하게

- 학생은 "할 숙제", "시험 보기", "내 성적 보기", "AI 튜터로 복기하기"에 집중합니다.
- 학생 화면에서는 교사용 검토 정보나 정답 노출이 섞이지 않아야 합니다.

## 3-1. 사용자 유형

- 추후 확장 가능: Admin, School Manager, Parent

## 3-2. Teacher 회원가입 요구사항

가입 흐름:
1. 교사 인증
2. Individual Settings 입력
3. Teaching Settings 입력
4. 회원가입 완료

Teacher Individual Settings:
- 로그인 정보
- 이름
- 전화번호
- 이메일

Teacher Teaching Settings:
- 담당 과목
- 담당 학년
- 담당 반

현재 합의된 기본안:
- 최초 1명은 `어드민 교사`로 부트스트랩합니다.
- 어드민 교사는 교사용 초대코드를 발급합니다.
- 이후 교사 회원가입은 기본적으로 `어드민 교사 초대코드`로만 진행합니다.
- 기존 학교 이메일 가입 요청은 예외적으로 어드민 화면에서 `pending / manual_review / verified` 로 관리합니다.

현재 구현 기준 추가 원칙:
- 최초 어드민 교사는 `bootstrap_admin` 검증 방식으로 기록됩니다.
- 어드민 교사만 `교사용 초대코드 발급` 과 `교사 승인 큐 관리` 권한을 가집니다.
- 일반 교사는 `학생 초대코드 발급` 과 수업 운영만 담당합니다.
- 교사 홈에는 여러 수업의 튜터 대화 원문을 묶은 `AI 브리핑` 영역이 함께 제공될 수 있어야 합니다.

## 3-3. Student 회원가입 요구사항

가입 흐름:
1. 학생 인증
2. Individual Settings 입력
3. 회원가입 완료

Student Individual Settings:
- 로그인 정보
- 이름
- 전화번호
- 이메일

학생 인증은 아직 확정 전이지만, 권장 흐름은 아래입니다.

권장 Student 인증 방식:
1. 교사가 발급한 반 초대코드 + 학생 번호
2. 이름 + 생년월일 일부 + 반 코드
3. 보호자 연락처 또는 학교 이메일 검증

추천 기본안:
- 교사/반 단위 초대코드
- 학생 번호 확인

현재 합의된 기본안:
- 학생 인증은 `반 초대코드 + 학생 번호`를 우선 적용합니다.

현재 구현 기준 추가 원칙:
- 학생 회원가입 검증은 `반 초대코드 + 학생 번호`를 사용합니다.
- 검증 완료된 교사는 반/수업 단위로 학생용 초대코드를 생성할 수 있어야 합니다.
- 학생이 초대코드로 가입하면 해당 반/수업에 자동 연결돼야 합니다.
- 승인 관리 화면에서는 `어드민 교사 전용 교사 초대코드 발급`, `pending / manual_review 교사 큐`, `승인 메모`를 함께 관리할 수 있어야 합니다.

## 3-4. 설정 수정 정책

- 회원가입 후에도 각 설정 항목은 재인증 후 수정 가능해야 합니다.
- 이메일, 전화번호, 담당 반 같은 민감한 정보는 별도 인증 과정을 거쳐 수정합니다.

## 4. Teacher 웹 기능 정의

## 4-1. Teacher Home

Teacher 로그인 후 첫 화면입니다.

좌측 사이드바:
- hover 또는 클릭으로 열림
- 다시 닫기 가능
- 하단에 `Settings` 버튼
- `승인 관리`
- `학생 관리`
- `과목 관리`
- `시험지 제작`
- `자료실`
- 추후 확장: `알림`, `보강수업 추천`, `반 관리`

Settings 팝업:
- Teacher Individual Settings
- Teacher Teaching Settings

가운데 메인 컨텐츠:
- 반별 진도 캘린더
- 교과별 진도 캘린더
- 교사 전체 수업 기준 AI 대화 브리핑

반별 진도 캘린더:
- 각 반이 어떤 시험/숙제까지 진행했는지 표시
- 교재 목차 기준으로 현재 진도를 보여줌
- 반마다 다른 진도를 달력과 타임라인으로 비교 가능
- 현재 구현은 시험/숙제의 배포일과 마감일을 기준으로 달력 보드를 먼저 제공하고, 이후 교재 목차 진도율을 더 정교하게 연결합니다.

교과별 진도 캘린더:
- 각 과목이 목차 기준으로 어디까지 진행됐는지 표시
- 0%~100% 진도율로 표현
- 현재 구현은 과목별 시험/숙제 일정 밀집도를 달력 보드로 먼저 보여주고, 이후 목차 진도율과 직접 연결합니다.
- 시험 배포/완료 이력과 연계

AI 대화 브리핑:
- 여러 수업의 소크라테스 튜터 대화 원문을 모아 공통 오개념, 질문 패턴, 교사용 말하기 문장을 요약합니다.
- 교사는 홈 화면에서 `executive summary`, `공통 오개념`, `질문 패턴`, `teacher talk track`, `개입 추천`을 함께 확인합니다.
- 브리핑에는 어떤 수업에서 나온 신호인지 연결되는 `수업별 위험 지도`가 함께 있어야 합니다.

## 4-2. 학생 관리 페이지

좌측 사이드바:
- 담당 반 리스트
- 예: 1반, 3반, 4반, 8반

특정 반 클릭 시:
- `OO반` 페이지 진입

OO반 페이지:
- 상단에 `반 진도 사항` 버튼
- 학생 번호 순 스크롤 리스트
- 특정 학생 클릭 시 `OOO 학생 관리` 페이지 이동

OO반 진도 사항 페이지:
- 조회할 교과 선택
- 교과 선택 후 목차별 평균 시험 성적을 막대그래프로 표시
- 기준 점수는 "최초 시험 점수"
- 재시도 후 최종 정답 도달 여부는 별도 지표로 분리

OOO 학생 관리 페이지:
- 교사 메모 노트
- 학생 개인 과목별 평균 시험 성적 그래프
- 과목별 목차 순서 그래프
- `AI 튜터 분석` 버튼

AI 튜터 분석:
- 담당 과목 리스트 표시
- 과목 선택 시 해당 학생의 소크라테스 대화 이력을 AI가 분석
- 잘 이해한 개념
- 이해가 부족한 개념
- 반복되는 오개념
- 복기에 잘 반응한 질문 방식
- 최근 원문 하이라이트와 세션별 핵심 질문 표시
- 보강수업 시 교사가 직접 말해줄 추천 포인트
- 학생 상세 화면 상단에는 위 신호를 요약한 `학생 개별 Gemini 브리핑`이 별도 카드로 보여야 합니다.

## 4-3. 과목 관리 페이지

- 담당 과목 리스트 표시
- 과목별 목차 점수 평균 표시
- 전체 반 평균과 반별 평균을 비교
- `반별 조회` 버튼으로 반 단위 분리
- 그래프 형식은 모두 같은 양식의 막대그래프 사용
- 선택한 과목에 대해 시험 결과와 튜터 대화를 함께 읽은 `과목별 Gemini 브리핑`이 같이 보여야 합니다.

## 4-4. 시험지 제작 페이지

이 페이지는 교사가 가장 자주 쓰는 핵심 작업 공간입니다.

좌측 사이드바 선택 흐름:
1. 과목 선택
2. 교재 또는 업로드 자료 선택
3. 목차 또는 자료 범위 선택
4. 세부 시험 초안 선택

동작 방식:
- 기본은 접힌 선택 버튼
- 버튼 클릭 시 리스트가 펼쳐짐
- 선택하면 해당 리스트는 닫히고 다음 리스트가 열림
- 목차까지 선택 완료되면 좌측 사이드바는 닫히고 작업 팝업 또는 전용 작업 공간이 열림

실시간 제작 상태 표시:
- 교재 파싱중
- 목차 구조화중
- 문제 제작중
- 해설 제작중
- 검토 준비중

현재 구현 기준 추가 원칙:
- 자료 업로드는 최소한 `queued -> parsing -> indexing -> completed/failed` 상태를 DB에 남겨야 합니다.
- 교사는 자료실에서 현재 처리 단계와 실패 사유를 다시 확인할 수 있어야 합니다.
- 자료실 상세 화면은 진행률 바와 `자료 등록 -> 파싱/인덱싱 -> 단원별 초안 생성 -> 교실 연결` 4단계 보드로 상태를 보여줘야 합니다.
- 업로드 자료는 페이지 단위 텍스트와 페이지 수를 DB에 남겨야 합니다.
- AI는 업로드 자료 요약, 단원 후보, 교육 목적 초안을 생성할 수 있어야 합니다.
- 자료 기반 자동 draft 시험은 `material_id`, `learning_objective`를 포함한 저장된 시험으로 이어져야 합니다.
- 시험지 제작실 안에서 `DB 교재 카탈로그`를 다시 동기화하고 교재/단원 구조를 바로 열람할 수 있어야 합니다.
- 교재 카탈로그의 각 단원은 현재 연결된 AI 생성 초안 개수를 함께 보여줘야 합니다.
- 단원에 이미 초안이 있으면 교사는 카탈로그 화면에서 바로 왼쪽 4단계 선택 흐름으로 점프할 수 있어야 합니다.

시험지 제작 완료 후 기능:
- 기본 10문항 생성
- 문제 수정
- 문제 삭제
- 문제 추가
- 정답 수정
- 해설 수정
- 교육 목적 수정

시험지 메타데이터:
- 과목
- 교재
- 목차
- 교육 목적
- 생성 일시
- 생성 모델
- 근거 페이지

교재 근거 UX:
- 문제별로 실제 PDF 교재 근거 페이지 링크를 가짐
- 여러 페이지면 여러 페이지 모두 연결
- 클릭 시 PDF 리더 팝업 또는 ebook 뷰어 팝업 표시
- 교사는 출제 중과 출제 후 모두 열람 가능

저장과 배포:
- 시험지를 draft로 저장
- 저장된 draft는 다시 열어서 문항/해설/근거를 계속 수정 가능
- 검수 완료 후 publish
- 예약 배포 가능
- 언제 공지되는지, 언제까지 수행해야 하는지 설정

현재 구현 메모:
- 예약 배포는 `workflow_status = scheduled` 와 `published_at = 시작 시각`으로 저장합니다.
- 학생/교사 화면이 시험 목록을 읽을 때 시작 시각이 지난 예약 시험을 자동 활성화하고 알림을 만듭니다.
- 현재는 외부 크론 없이 동작하는 lazy activation 방식이며, 이후 전용 스케줄러로 확장 가능합니다.
- `009_textbook_catalog.sql` 적용 후에는 시험지 제작실에서 `textbooks / textbook_toc_nodes / textbook_pages` 기반 교재 카탈로그를 바로 열람할 수 있습니다.
- 마이그레이션이 아직 없더라도 읽기 전용 카탈로그 열람은 filesystem fallback으로 동작하고, 전체/개별 동기화 버튼은 마이그레이션 적용 후 활성 의미를 가집니다.
- 동기화된 교재 범위에서 새 시험을 저장하거나 기존 시험을 수정하면, 해당 시험 row에 `textbook_id / textbook_toc_node_id`를 함께 기록해 이후 분석과 교재 추적에 사용합니다.
- 교사는 선택한 교재에 대해 이미 저장된 기존 시험의 `textbook_id / textbook_toc_node_id` 연결을 다시 보정하는 backfill을 실행할 수 있어야 합니다.
- `010_exam_question_chunk_refs.sql` 적용 후에는 문항별 `source_chunk_ids`를 저장하고, 학생/교사 화면에서 페이지 이미지와 함께 근거 텍스트 조각을 바로 볼 수 있어야 합니다.

운영 안전 규칙:
- 학생 응시 기록이 생긴 시험은 원본 문항 수정을 잠급니다.
- 응시 후 수정이 필요하면 기존 시험을 보관하고 새 draft를 만들어 다시 배포하는 흐름을 기준으로 합니다.

공지 후 학생 UX:
- 학생 로그인 시 시험 알람 표시
- 시험 완료 전까지 알람 유지

### 운영 단위 결정

- 현재 시스템의 최상위 운영 단위는 `course` 입니다.
- 현 단계에서 `course` 는 단순한 "반"도, 단순한 "과목"도 아닙니다.
- `course` 는 `교사 x 반 x 과목` 조합으로 구성된 "담당 수업 운영 단위"로 정의합니다.
- 예: `중1-3반 수학`, `중1-3반 국어`, `중2-1반 과학`
- 이렇게 두는 이유는 현재 코드에서 자료 업로드, 시험, 응시, 분석, 튜터 로그가 모두 `course_id` 기준으로 묶여 있기 때문입니다.
- 단기 구현에서는 기존 `courses` 테이블을 이 의미로 유지합니다.
- 중기적으로는 `school_classes`, `subjects` 같은 정규화 엔티티를 추가해도, 운영 루트 키는 계속 `course` 로 유지할 수 있습니다.
